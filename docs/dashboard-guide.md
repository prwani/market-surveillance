# Market Surveillance Dashboard — User Guide

## Overview

The Market Surveillance Dashboard is a browser-based interface for detecting and
investigating market manipulation across Asian exchanges (SGX, HKEX, NSE). It is
built for compliance analysts, surveillance officers, and exchange regulators who
need to monitor trading activity, review alerts generated by AI detection agents,
and manage intervention cases.

The dashboard is powered by a FastAPI backend with five AI agents that analyze
simulated or live exchange data to detect spoofing, layering, wash trading, price
anomalies, and coordinated cross-market manipulation.

## Accessing the Dashboard

### Local Development

```bash
uvicorn app.main:app --host 0.0.0.0 --port 8080 --reload
```

Open [http://localhost:8080](http://localhost:8080) in your browser.

### Azure Deployment

After deploying with `azd up`, the dashboard is available at the URL
printed at the end of deployment. The URL follows this pattern:

```
https://<app-name>.<environment-id>.<region>.azurecontainerapps.io
```

No login is required — the dashboard is open by default. Add authentication
via the Azure Container Apps ingress settings if needed.

## Dashboard Pages

### 1. Home (`/`)

The landing page shows a real-time overview of the surveillance system.

**Summary cards** display four key metrics:

| Card | Meaning |
|---|---|
| **Total Events** | Number of market events (trades, order book updates) processed |
| **Active Alerts** | Number of alerts generated by detection agents |
| **Open Cases** | Number of intervention cases created for high-confidence alerts |
| **Reports Generated** | Number of evidence reports compiled for cases |

**Quick Actions** provide one-click navigation:

- **Run Simulation** → go to the `/simulate` page
- **View Alerts** → go to the `/alerts` page
- **View Cases** → go to the `/cases` page
- **KQL Query** → go to the `/kql` page

> **First time?** All counters start at zero. Navigate to **Run Simulation** to
> generate test data and see the system in action.

### 2. Run Simulation (`/simulate`)

The simulation page lets you generate synthetic exchange data with injected
manipulation patterns. This is the primary way to test the detection pipeline
without connecting to live market feeds.

#### Step-by-step

1. **Exchanges** — Enter one or more exchange codes separated by commas.
   Supported exchanges: `SGX` (Singapore), `HKEX` (Hong Kong), `NSE` (India).
   Example: `SGX,HKEX`

2. **Duration** — Set the number of seconds of trading activity to simulate.
   A 120-second simulation generates roughly 10,000–20,000 events depending on
   the number of exchanges.

3. **Manipulation types** — Check the boxes for the manipulation patterns to
   inject into the simulated data:
   - **Inject Spoofing** — large orders placed and rapidly cancelled
   - **Inject Layering** — orders at multiple price levels then mass-cancelled
   - **Inject Wash Trading** — same entity on both sides of trades
   - **Inject Price Anomaly** — unusual price movements injected statistically

4. **Click "▶ Run Simulation"** — The simulation runs server-side. This may
   take 10–30 seconds depending on duration and exchange count.

5. **Results** — When complete, a summary shows:
   - `event_count` — total market events generated
   - `alert_count` — alerts produced by the detection agents
   - `case_count` — intervention cases created for high-confidence alerts
   - `report_count` — evidence reports compiled
   - `eventhouse_rows` — rows ingested into Fabric Eventhouse (0 if not connected)

#### What happens behind the scenes

```
Simulator → generates events → Pattern Agent detects spoofing/layering
                              → Anomaly Agent detects price/volume anomalies
                              → Cross-Market Agent detects coordinated manipulation
                              → Intervention Agent creates cases (confidence > 0.70)
                              → Evidence Agent compiles forensic reports
```

### 3. Alerts (`/alerts`)

The alerts page shows a table of every alert generated by the detection agents.

#### Columns

| Column | Description |
|---|---|
| **ID** | Unique alert identifier (e.g., `SPOOF-15608D`, `LAYER-8F7512`, `WASH-35BB8C`) |
| **Type** | The manipulation pattern detected (see table below) |
| **Severity** | Alert severity level with color-coded badge |
| **Exchange** | Exchange where the activity was detected (SGX, HKEX, NSE) |
| **Symbol** | The instrument/ticker involved (e.g., DBS, OCBC, 0700.HK) |
| **Confidence** | Detection confidence as a percentage (0–100%) |
| **Time** | ISO 8601 timestamp when the alert was generated |

#### Alert Types

| Type | Prefix | Description |
|---|---|---|
| **SPOOFING** | `SPOOF-` | Large orders placed then quickly cancelled to manipulate the price. The agent looks for high cancel rates (>80%) with short cancel latency (<500ms) on large orders. |
| **LAYERING** | `LAYER-` | Multiple orders placed at different price levels to create false depth in the order book, then mass-cancelled. Detected by analyzing orders across multiple price levels from the same broker. |
| **WASH_TRADING** | `WASH-` | The same entity (or linked accounts) appears on both sides of trades to inflate volume artificially. Detected when buyer and seller broker IDs are linked or identical. |
| **PRICE_ANOMALY** | `ANOM-` | Statistically unusual price or volume movements that deviate significantly from rolling averages. Detected by the Anomaly Detection Agent. |
| **COORDINATED_MANIPULATION** | `COORD-` | Correlated manipulation activity detected across multiple exchanges simultaneously. Identified by the Cross-Market Agent tracking symbol aliases. |

#### Severity Levels

| Severity | Badge Color | Meaning |
|---|---|---|
| **CRITICAL** | 🔴 Red | Immediate attention required — high-confidence, high-impact manipulation |
| **HIGH** | 🟠 Orange | Significant manipulation pattern detected |
| **MEDIUM** | 🟡 Yellow | Moderate suspicion — may require investigation |
| **LOW** | 🟢 Green | Minor anomaly — informational, review if recurring |

#### Confidence Score

The confidence score (0.0–1.0, displayed as 0–100%) indicates how strongly the
detection agent believes the activity is genuine manipulation:

- **> 0.70** — High confidence. An intervention case is automatically created.
- **0.40–0.70** — Moderate confidence. Logged as an alert for manual review.
- **< 0.40** — Low confidence. Informational; may indicate noise.

### 4. Cases (`/cases`)

The cases page lists intervention cases — automated responses triggered when an
alert exceeds the confidence threshold (default: 0.70).

#### Columns

| Column | Description |
|---|---|
| **Case ID** | Unique case identifier (e.g., `CASE-2C2D904`). Click to view the evidence report. |
| **Status** | Current case status with color-coded badge |
| **Alert Type** | The manipulation type that triggered the case |
| **Exchange** | Exchange where the manipulation was detected |
| **Symbol** | The instrument/ticker involved |
| **Actions** | Automated actions taken (e.g., halt trading, notify regulator) |

#### Case Statuses

| Status | Badge Color | Description |
|---|---|---|
| **OPEN** | 🔵 Blue | Case created, initial investigation in progress |
| **HALTED** | 🔴 Red | Trading has been halted for the affected symbol |
| **NOTIFIED** | 🟠 Orange | Regulatory authority has been notified |
| **CLOSED** | 🟢 Green | Investigation complete, case resolved |
| **ESCALATED** | 🟣 Purple | Case escalated for further review by senior compliance |

#### Automated Actions

When a case is created, the Intervention Agent may take the following actions
(executed in dry-run mode during simulations):

- **HALT_TRADING** — Sends a halt request to the exchange API for the affected
  symbol. In simulation mode the request is accepted but not executed on a live
  exchange.
- **NOTIFY_REGULATOR** — Sends a notification to the relevant regulatory body
  (e.g., MAS for SGX, SFC for HKEX, SEBI for NSE) with case details and
  preliminary evidence.
- **SUSPEND_BROKER** — Suspends the implicated broker's access pending
  investigation (used in severe cases).

### 5. Evidence Reports (`/reports/{case_id}`)

Click any **Case ID** on the Cases page to view the full evidence report.

Each report contains:

| Section | Description |
|---|---|
| **Report header** | Report ID, case ID, and generation timestamp |
| **Overview cards** | Exchange, symbol, manipulation type, regulatory body, involved entities, and evidence time window |
| **Statistics** | Price impact (%), volume affected, estimated gain ($), and per-pattern scores (spoofing, layering, anomaly) |
| **Narrative** | AI-generated executive summary including case details, involved entities, market impact analysis, and a timeline reconstruction |
| **Related Order Events** | Raw order book events within the evidence window, showing each event's timestamp, action, side, price, quantity, and broker ID |
| **Related Trades** | Trade records within the evidence window with buyer/seller details |

Evidence reports are designed for regulatory submission. The narrative section
provides a human-readable summary suitable for compliance filings with
regulators such as MAS, SFC, or SEBI.

### 6. KQL Query Runner (`/kql`)

The KQL (Kusto Query Language) page provides a query editor for running ad-hoc
queries against the Fabric Eventhouse database.

> **Prerequisite:** The `KQL_URI` environment variable must be set to the Fabric
> Eventhouse endpoint. Without it, the KQL page returns an error.

#### How to use

1. Type a KQL query in the text area.
2. Click **"▶ Execute"**.
3. Results appear below the editor.

#### Example Queries

**Count total trades:**

```kql
TRADES
| count
```

**Find wash trade suspects:**

```kql
TRADES
| where buyer_id contains "WASH"
| take 10
```

**Spoofing detection (from `kql/spoofing_detection.kql`):**

```kql
ORDER_BOOK_EVENTS
| where event_time > ago(1m)
| where action in ("add", "cancel")
| summarize
    orders_added     = countif(action == "add"),
    orders_cancelled = countif(action == "cancel"),
    avg_size_added   = avgif(quantity, action == "add")
    by broker_id, symbol, exchange_id, bin(event_time, 1m)
| where orders_added > 0
| where (orders_cancelled * 1.0 / orders_added) > 0.80
| where avg_size_added > 10000
| order by orders_cancelled desc
```

**Recent order book activity for a symbol:**

```kql
ORDER_BOOK_EVENTS
| where symbol == "DBS"
| order by event_time desc
| take 50
```

The full set of production KQL queries is in the `kql/` directory:
`spoofing_detection.kql`, `layering_detection.kql`, `wash_trading_detection.kql`,
and `anomaly_detection.kql`.

## API Reference

For automation, integration, or scripting, use the JSON API directly.

| Method | Endpoint | Description |
|---|---|---|
| `POST` | `/api/simulate` | Run a simulation. Body: `{"exchanges":["SGX","HKEX"],"duration":120,"inject_spoofing":true,...}` |
| `GET` | `/api/alerts` | List all alerts as JSON |
| `GET` | `/api/cases` | List all intervention cases as JSON |
| `GET` | `/api/reports/{case_id}` | Get evidence report for a case |
| `GET` | `/api/stats` | System statistics: `{"total_events":N,"total_alerts":N,"total_cases":N,"total_reports":N}` |
| `POST` | `/api/kql` | Execute a KQL query. Body: `{"query":"TRADES | count"}`. Requires `KQL_URI`. |
| `GET` | `/healthz` | Health check |
| `GET` | `/ready` | Readiness check (includes KQL connection status) |

### Example: Run a Simulation via API

```bash
curl -X POST http://localhost:8080/api/simulate \
  -H "Content-Type: application/json" \
  -d '{
    "exchanges": ["SGX", "HKEX"],
    "duration": 120,
    "inject_spoofing": true,
    "inject_layering": true,
    "inject_wash_trading": true,
    "inject_price_anomaly": false
  }'
```

## Typical Workflow

1. **Run a simulation** — Go to `/simulate`, select exchanges and manipulation
   types, and click Run. This populates the system with test data.

2. **Review alerts** — Navigate to `/alerts`. Sort by severity to focus on
   CRITICAL and HIGH alerts first. Check confidence scores to distinguish
   strong signals from noise.

3. **Investigate cases** — Go to `/cases`. Cases are automatically created for
   alerts with confidence > 0.70. Click a Case ID to view the evidence report.

4. **Review evidence reports** — On the report page, read the narrative for a
   human-readable summary. Check the statistics for market impact. Review raw
   events for detailed forensic evidence.

5. **Run KQL queries** — Use `/kql` for deeper investigation. Query the raw
   event tables to find patterns not covered by automated detection.

6. **Iterate** — Run additional simulations with different parameters to test
   detection coverage. Adjust manipulation types and duration to stress-test
   the agents.

## Troubleshooting

| Problem | Cause | Solution |
|---|---|---|
| Dashboard shows 0 alerts / 0 cases | No simulation has been run yet | Go to `/simulate` and run a simulation |
| KQL page shows an error | `KQL_URI` environment variable not set | Set `KQL_URI` to your Fabric Eventhouse endpoint and restart the app |
| Simulation takes a long time | Large duration or many exchanges | Reduce duration (try 60s) or use fewer exchanges |
| No cases created after simulation | All alerts below confidence threshold | Enable more manipulation types (especially wash trading, which tends to produce high-confidence alerts) |
| Report page shows 404 | Case ID not found or typo | Copy the exact Case ID from the `/cases` page |
| `eventhouse_rows: 0` after simulation | Not connected to Fabric Eventhouse | This is expected in local mode. Events are processed in-memory by the agents. Fabric ingestion requires a deployed Eventhouse with Eventstreams configured. |
